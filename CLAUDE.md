# バイオインフォマティクス講義ノート - プロジェクトコンテキスト

## 📝 プロジェクト概要

CourseraのBioinformatics Specializationの講義ノートをDocusaurusで管理するプロジェクト。
講義の内容を日本語でまとめ、アルゴリズムの実装例と共に学習できるサイトを構築中。

### 🌐 デプロイ情報

- URL: https://susumutomita.github.io/bioinformatics-note/
- GitHub Pages + GitHub ActionsでのCI/CD構築済み
- mainブランチへのプッシュで自動デプロイ

## 🛠️ 技術スタック

- **フレームワーク**: Docusaurus v3.8.1
- **パッケージマネージャー**: Bun
- **CI/CD**: GitHub Actions
- **デプロイ**: GitHub Pages
- **ダイアグラム**: Mermaid（@docusaurus/theme-mermaid）
- **品質管理**:
  - textlint（日本語チェック）
  - markdownlint
  - prettier（フォーマット）
  - husky（pre-commitフック）

## ⚠️ 開発ルール（必須）

### 品質チェックの徹底

**すべてのコンテンツ追加・修正時に必ず実行**：

```bash
# 必ず実行して全てパスすること
make before-commit
```

エラーが出た場合は修正してから再度実行。
リント・フォーマットエラーは自動修正可能：

```bash
make lint-text-fix
make lint-markdown-fix
make format
```

### コンテンツ作成方針

1. **正確性の確保**
   - 情報の正確性を最優先
   - 不確かな情報は「推測」「仮説」と明記
   - 出典・参考文献を明記

2. **初学者への配慮**
   - ソフトウェアエンジニアがバイオインフォマティクスに取り組む際の視点を重視
   - 生物学の前提知識が必要な箇所は必ず説明
   - 躓きやすいポイントを省略せず丁寧に解説

3. **視覚的理解の促進**
   - 図・図解を積極的に使用（Mermaid図を活用）
   - コード例と実行結果を必ず示す
   - ステップバイステップの説明

4. **実用性の重視**
   - 理論だけでなく実装方法も詳細に説明
   - エラー処理やエッジケースも解説
   - 計算量や実行時間の考察を含める

### 📝 細野真宏スタイル執筆ガイドライン（必須・全記事適用）

#### 🔴 重要：今後作成する全ての講義ノートは、以下の細野真宏スタイルで執筆すること

#### 基本理念

「**読者が一度も躓かずに最後まで読み切れる**」を最優先目標とする。専門知識ゼロの読者でも、読み終わったときには専門家レベルの理解に到達できる構成を目指す。

#### 必須の文書構造（テンプレート）

```markdown
# [タイトル]（超詳細版）

## 🎯 まず、この講義で何を学ぶのか

最終ゴール：**[一文で明確に]**

でも、ちょっと待ってください。そもそも[基本的な疑問]？
実は、[驚きの事実や価値]なんです。

## 🤔 ステップ0：なぜ[テーマ]が重要なの？

### 0-1. そもそもの問題を考えてみよう

[身近な例や現実の問題から導入]

### 0-2. 驚きの事実

[なぜこの手法が画期的なのか]

## 📖 ステップ1〜10：[段階的な説明]

### 各ステップの構成

- まず素朴な疑問から
- なぜそうなるのか
- 具体例で確認
- ここがポイント！
```

#### 執筆時の鉄則（10箇条）

1. **疑問の先回り**
   - 読者が抱くであろう疑問を「でも待って...」で先回り
   - 「なぜ？」「どうして？」「本当に？」を多用

2. **アナロジーの徹底活用**
   - 新概念は必ず日常例で説明（例：「しりとり」「迷路」「料理のレシピ」）
   - 「これは〜のようなもの」を積極的に使用

3. **ステップの細分化**
   - 1つのステップで1つの概念のみ
   - 前のステップの理解を前提に積み上げ
   - 飛躍があったら即座に分割

4. **視覚的理解の促進**

   ```python
   # 必ずコメントで「なぜこうするか」を説明
   def example():
       """
       この関数が何をするか
       なぜ必要か
       どう使うか
       """
   ```

   - mermaid図は複雑にしすぎない
   - ASCIIアートも効果的に使用

5. **「魔法」の解明**
   - 「魔法のような」「不思議な」は必ず仕組みを説明
   - ブラックボックスを作らない
   - 「実は...」で種明かし

6. **実験・体験の重視**

   ```python
   def experiment_with_concept():
       """
       読者が自分で試せる例を提供
       """
       print("ケース1：こうすると...")
       print("ケース2：こうすると...")
       print("パターンが見えてきた！")
   ```

7. **失敗例の提示**
   - 「普通はこう考えるけど...」
   - 「これだとうまくいかない、なぜなら...」
   - 「だから、こういう工夫が必要」

8. **段階的な理解レベル**

   ```markdown
   ## 📝 まとめ：今日学んだことを整理

   ### レベル1：表面的理解（これだけでもOK）

   ### レベル2：本質的理解（ここまで来たら素晴らしい）

   ### レベル3：応用的理解（プロレベル）
   ```

9. **感嘆と発見の演出**
   - 「驚きの事実」「ここで天才的な発想が！」
   - 「なんと！」「実は...」
   - 読者の「なるほど！」を引き出す

10. **次回への期待感**
    - 「さらに驚くべき...」
    - 「次回は〜の謎に迫ります」
    - 具体的な予告で興味を持続

#### 特に重要な表現パターン

```markdown
# よく使う接続詞・フレーズ

- でも、ちょっと待ってください
- そもそも〜って何でしょう？
- ここで重要な観察をしてみましょう
- 実験してみましょう
- つまり、言い換えると...
- なぜこうなるのか？それは...
- ここが「魔法」のような部分です
- でも実は、種を明かすと...
```

#### チェックリスト（公開前に必ず確認）

- [ ] 専門用語は初出時に必ず説明されているか
- [ ] 各ステップに飛躍はないか
- [ ] 具体例は十分か（最低3つ）
- [ ] 図解は理解を助けているか
- [ ] コードにコメントは十分か
- [ ] 「なぜ」にすべて答えているか
- [ ] 読者の疑問を先回りできているか
- [ ] まとめは3段階になっているか
- [ ] 次回予告はワクワクするか

#### 文字数の目安

- 通常版：2000-3000文字
- 詳細版（-detailed.md）：5000-10000文字
- 各ステップ：300-500文字
- 具体例1つ：100-200文字

## 💡 重要な設計判断

1. **日本語優先**
   - Courseraの英語講義を日本語で解説
   - 専門用語は英語併記

2. **コード例の充実**
   - Pythonでの実装例を豊富に提供
   - 段階的な改良版も掲載

3. **生物学的背景の重視**
   - アルゴリズムだけでなく、生物学的意味も詳しく説明
   - 実際のゲノムでの例を積極的に使用
